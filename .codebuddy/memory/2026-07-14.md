## code-server 原生模块缺失警告修复

### 现象
`cargo check`/`cargo build` 时 build.rs 报 `6 native module(s) missing from code-server`（winregistry/deviceid/watchdog/spdlog/sqlite3/crypt32），仅 windows-process-tree 在。这是 warning 非 error，不影响主程序，只影响内置 IDE。

### 根因
- code-server 的 7 个 `@vscode` 原生模块需本地 C++ 编译。`build.rs` 用 `npm install --production --ignore-scripts` 故意不编；真正编译在 `scripts/download-code-server.mjs` 末尾的 `npm install --production`（lib/vscode）。
- 本机 VS 2026 Insiders 未装 Spectre 缓解库，而各 `*.gyp` 设了 `SpectreMitigation: Spectre` → node-gyp 编译失败（MSB8040）→ `.node` 缺失。
- 只有 sqlite3 额外有 `deps/sqlite3.gyp`（编译 SQLite amalgamation），其 `build/deps/*.vcxproj` 也带 Spectre，所以只有它中招最顽固。
- "旧版本没有"：以前这些 `.node` 在某次成功编译后存在（或 Spectre 库当时在）；现在 code-server 被重新解压成干净版需重编，当前环境缺 Spectre 库故报缺失。

### 修复（已固化到 download-code-server.mjs）
1. 新增 `stripSpectreMitigation()`：**递归**扫描 `@vscode/**/*.gyp`（含 `deps/`），删除 `SpectreMitigation: Spectre` 设置，在 `npm install --production` 编译前调用。
2. 立即对磁盘上文件做了同样处理并重编：
   - 用 node 脚本 patch 7 个 `binding.gyp`（windows-process-tree 是内联写法需放宽正则）。
   - `npm rebuild` 在 lib/vscode 会因无关依赖 `kerberos` 编译失败而中断，只编出 6/7。
   - 单独对 `sqlite3` 目录 `node-gyp rebuild`：先 patch `deps/sqlite3.gyp` 并 `Remove-Item build/` 清缓存再编，成功生成 `vscode-sqlite3.node`。
3. 验证：`cargo check` 日志已无任何 "native module" 警告，`Finished` 通过。

### 教训
- `npm rebuild` 在 lib/vscode 会因个别无关原生依赖（kerberos 等）失败而中断，导致 @vscode 模块编不全；应逐个对缺失模块 `node-gyp rebuild`。
- 后台长编译用 `Start-Process -FilePath cmd.exe -ArgumentList "/c ..."`（npm 是 .cmd，Start-Process 不能直接启 npm），日志重定向到文件后轮询。

## 排查：skills/mcp GitHub 源连不上 = hosts 劫持（2026-07-14）

现象：`[skills] GitHub 搜索失败: error sending request for url (https://api.github.com/...)`，但 ClawHub 成功。

根因（非代码问题）：本机 `C:\Windows\System32\drivers\etc\hosts` 把整个 GitHub 域名族劫持到 `127.0.0.1`（github.com / api.github.com / raw.githubusercontent.com / githubusercontent.com 等共 27 条）。
- `curl.exe -v https://api.github.com` 显示 `IPv4: 127.0.0.1` → `Trying 127.0.0.1:443` → schannel 失败，坐实是 DNS/hosts 劫持。
- 连带影响：skills/mcp/plugins 的 GitHub 在线市场、`fetch_local_market`（raw.githubusercontent.com）、以及之前的 `git push` 443 超时，全是同一原因。
- ClawHub 不受影响是因为域名不在劫持名单。

解决（需管理员权限，hosts 在 workspace 外，不擅自改）：管理员 PowerShell 备份 + 删除所有 `127.0.0.1 *github*` 行 + `ipconfig /flushdns`：
`$h="$env:windir\System32\drivers\etc\hosts"; Copy-Item $h "$h.bak" -Force; (Get-Content $h) | Where-Object { $_ -notmatch '^\s*127\.0\.0\.1\s+.*github' } | Set-Content $h -Encoding ASCII; ipconfig /flushdns`
验证：`Resolve-DnsName api.github.com`（应不再是 127.0.0.1）。
教训：市场类模块「连不上」先用 curl -v 看 DNS 解析到哪，优先排查 hosts/代理，别改代码。

`mcp.rs` 的 `resolve_one()` 使用 `std::fs::canonicalize()` 解析相对路径为绝对路径。Windows 上 `canonicalize()` 返回 `\\?\C:\...` 格式的扩展路径（verbatim prefix），`to_string_lossy()` 不会去掉此前缀。Node.js v24 无法识别 `\\?\` 路径（EISDIR: illegal operation on a directory, lstat 'c:'），导致 `node \\?\c:\...\index.mjs` 进程立即退出，Rust 端在 initialize 握手时读到 EOF → MCP-002 → 被包装为 MCP-011。

### 修复
`mcp.rs` 的 `resolve_one()` 中两处 `canonicalize()` 返回值新增 `strip_verbatim_prefix()` 处理：在 Windows 上用 `strip_prefix(r"\\?\")` 去掉前缀，非 Windows 原样返回。与 `code_server.rs:272` 已有的同类修复一致。

### 验证
- `cargo check` 通过，零新错误
- 手动测试 `node \\?\c:\...\index.mjs` 确认会触发 EISDIR，`node c:\...\index.mjs` 正常工作

## 修复：agent 多轮循环中「中间推理草稿」污染最终答案（2026-07-14）

### 问题（用户报告）
深度思考与流式回答没有区分"哪里是内部思考、哪里是给用户的正式答案"。

### 根因
`agent_loop.rs` 的 `run_agent_loop` 是多轮循环：每轮调 `run_completion`，带 `tool_calls` 就执行工具进下一轮，`tool_calls` 为空才是最终答案。但 `run_completion`（lib.rs）对**每一轮**都把普通 `content` 以 `stream-token` 推到答案框。于是**中间轮的推理草稿**（"我要调用xxx工具"）也被写进 `msg.content`，和最终答案糊在同一个回答气泡里。注：`reasoning_content`（DeepSeek/Claude 思考链）走 `thinking-*` 事件本来就是对的，不用动。

### 网上做法（已查证，2篇）
- CSDN《Ollama Streaming/Thinking/Tool Calling》：thinking/content/tool_calls 是 API 三个独立字段，分别累积——已做到。
- 博客园《深入理解 Agent Loop》：**关键缺失环节**——业界共识是「只有 Loop 终止轮（不再请求工具）返回的纯文本才是用户答案」，中间轮文本是内部推理不暴露。Claude SDK：AssistantMessage(中间) vs ResultMessage(终态)；OpenAI SDK：`result.final_output` 才是最终答案。

### 方案（用户确认采用"回放最终答案"）
核心：agent 模式下 content 先静默缓冲，待判定「中间轮 vs 终止轮」后再回放归属。
1. **lib.rs**：`run_completion` 加参数 `stream_content: bool`。true(chat模式)=content 实时 `stream-token`；false(agent模式)=content 仅累积不 emit。`reasoning_content` 始终实时 `thinking-delta`（不变）。新增 `FinalAnswerStart` 事件结构体。
2. **agent_loop.rs**：`LlmClient::complete` trait + `RealLlmClient` + `llm_complete_retried` + `FakeLlmClient`(测试) 全部加 `stream_content` 参数透传。主循环：`stream_content = !agent_mode`；agent 模式下——终止轮(无 tool_calls) → emit `final-answer-start` + content 回放为 `stream-token`(最终答案) + `stream-done`；中间轮(有 tool_calls) → content 回放为 `thinking-start/delta/stop`(推理草稿进思考面板) 再执行工具。
3. **ChatView.tsx**：新增 `final-answer-start` 监听(进入答案阶段 setIsThinking(false))并在末尾 unlisten。其余零改动——中间轮 content 经 thinking-delta 天然进思考面板，终止轮经 stream-token 天然进答案框。

### 权衡
最终答案采用"回放"(整段瞬时出现，非逐字流式)，因为是否终止要等本轮流结束才知道。reasoning_content 思考链仍实时流式，实时体验不受影响。

### 验证
- 后端 `cargo check` 通过、`read_lints` 零错误、`cargo test --lib agent_loop` 7/7 全过。
- 前端 `read_lints` 零错误。

### P3：MessagePart 统一渲染架构

**目标**：ChatView 不再内联渲染 thinking/tool/content，改为通过 Part 注册表模式委托给独立渲染器。

**架构**：
```
Message (store) + 流式上下文
        │
        ▼
  messageToParts()  ← 桥接：扁平字段 → [MessagePart]
        │
        ▼
  MessageContent ← 遍历 parts → PartRenderer
        │
  ┌─────┼─────┬──────────┐
  ▼     ▼     ▼          ▼
Think Tool  Content   (未来:ImageGen, FilePreview...)
```

**新增文件**：
1. **`src/components/chat/types.ts`**（+95 行）：
   - `MessagePart = ThinkingPartData | ContentPartData | ToolCallPartData` 联合类型
   - `thinking` part：`{ type, text, state: "streaming"|"done", stats? }`
   - `content` part：`{ type, text }`
   - `tool_call` part：`{ type, steps: ToolStep[] }`
   - `messageToParts()` 桥接函数：将 Message 扁平字段 + 流式上下文（streamingThinking/isThinking等）统一转换为 Part 列表，ChatView 无需关心转换逻辑
