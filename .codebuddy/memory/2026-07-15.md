## Agent Loop 网页访问修复（2026-07-15）

### 问题诊断
用户报告 agent loop 给链接无法正常访问。经排查发现三个关键问题：

1. **缺失原生网页抓取工具**：10 个原生工具中无 `web_fetch`，全靠 MCP 的 `fetch_url`。如果 MCP web server 不工作，LLM 无法访问任何 URL。

2. **Agent loop 最终答案不回显**：`run_agent_loop` 中 `stream_content=false` 导致 LLM 响应完全缓冲，最终答案作为单个巨大 `stream-token` 事件一次性发出（而非逐字流式）。用户体验：长等待 → 突然全部出现。

3. **MCP `fetch_url` 不够健壮**：无超时、无 HTTP 状态检查、无 URL 验证、不支持 JSON/XML 内容类型。

### 修复方案（保持去耦合）

**Fix 1 — `tools.rs`：新增 `native_web_fetch` 工具**
- 使用 `reqwest::Client`（项目已有依赖），不依赖 MCP
- 30s 超时、HTTP 状态码检查、HTML 标签剥离 + 实体解码 + 空白压缩
- 支持 HTML/text/JSON/XML 内容类型
- 结果截断到 `max_chars`（默认 8000）
- 位于 `default_native_tools()` 中第 11 位，通过 ToolRegistry 统一管理

**Fix 2 — `agent_loop.rs`：修复最终答案流式回显**
- 新增 `stream_final_answer()` 函数：10 字符/块 + 10ms 延迟，模拟真实 LLM 打字
- 替代原来的一整块 `stream-token` 发射
- 中间轮推理仍走 `stream_content_as_thinking` → thinking-delta

**Fix 3 — `mcp-servers/web/index.mjs`：增强 `fetch_url` 健壮性**
- 新增 `fetchWithTimeout` 封装（AbortController，30s 超时）
- `fetch_url`：URL 格式验证（new URL 解析）、HTTP 状态检查、支持 JSON/XML
- `web_search`：增加 HTTP 状态检查
- 统一 User-Agent → Votek/0.3，添加 Accept/Accept-Language 头
- 更好的错误消息（区分超时/连接/HTTP 错误）

### 编译结果
`cargo check` 通过，仅 1 个已有 warning（McpToolExecutor 未使用）。

## Agent Loop "不回复" 诊断与加固（2026-07-15 第二轮）

### 用户报告
说话完全不理，无任何回复。

### 排查结果（cargo check 通过，无编译错误）

**已修复的潜在问题：**

1. **HTTP_CLIENT 无超时（最可能根因）**：LazyLock 创建的 reqwest Client 没有任何 timeout，如果网络慢/API 挂起，请求会无限等待。已添加 `connect_timeout=10s` + `read_timeout=120s`。

2. **LLM 返回空内容时静默**：当 LLM 返回 `content=""` 且 `tool_calls=[]` 时，Agent Loop 只发出 `final-answer-start` + `stream-done`，无任何文本。用户看到的是"发送了消息但没有任何回复"。已添加 fallback 文本告知用户。

3. **双重 stream-done**：`run_agent_loop` 返回后 `cleanup()` 又 emit 一次 `stream-done`，可能导致前端状态混乱。已去掉 `cleanup` 中的多余 emit。

4. **全链路诊断日志**：在 `chat_stream`、`prepare_loop_messages`、`run_completion` HTTP 请求/响应、Agent Loop 每轮 LLM 调用处添加了 `eprintln!` 日志。

**修改文件：**
- `lib.rs`：HTTP_CLIENT 超时 + chat_stream/prepare/run_completion 日志 + cleanup 去 stream-done
- `agent_loop.rs`：Agent Loop 启动/每轮 LLM/错误 日志 + 空响应 fallback

### 给用户的排查指引
后端日志会通过 stderr 输出，Tauri dev 模式下在终端可见。关键日志格式：
- `[chat_stream] 收到请求` → 前端调用成功
- `[LLM] POST ...` → 开始 HTTP 请求
- `[LLM] 响应状态: 200` → API 返回正常
- `[AgentLoop] LLM 返回 content=X chars tool_calls=Y` → LLM 响应详情
- 如果只看到 `收到请求` 但没有 `POST` → `prepare_loop_messages` 或 tools 获取卡住
- 如果只看到 `POST` 但没有 `响应状态` → 网络挂起（timeout 修复应解决）
- 如果看到 `响应状态` 不是 200 → API Key/配额问题

## Windows MCP 自动化集成 — sbroenne/mcp-windows v1.3.16

### 方案决策
选择 sbroenne/mcp-windows (Rust standalone .exe) 而非 CursorTouch/Windows-MCP (Python 3.13+)：
- 零依赖：单个 exe，无需 Python/.NET/任何运行时
- 技术栈对齐：Rust 后端，Votek 团队维护/理解成本低
- 语义 UI 定位：按名称/类型/ID 定位控件，不靠坐标（确定性交互，不受 DPI/主题影响）
- 内置 OCR：Windows.Media.Ocr 回退
- LLM token 优化：字段压缩 60%
- 54 个自动化测试，GPT-4.1/5.2 验证

作为内置 MCP Server（stdio+JSON-RPC），与 web/tavily 同层：
- 配置驱动：通过 McpServerConfig 动态连接，零 Rust 代码修改
- 工具通过 tools/list 动态发现，ToolRegistry 统一调度
- 工具以 `windows::ui_find` / `windows::screenshot` 等格式暴露给 LLM

### 解耦设计
- 二进制可选：未下载时连接静默失败，不影响核心应用
- 平台无感：非 Windows 上路径不存在自动跳过
